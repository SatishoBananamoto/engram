# AI agent credentials are readable by the agent itself — no platform solves this

## Context

During kv-secrets security testing (2026-03-21), we tested whether an AI agent can read not just user secrets (kv vault) but also its OWN platform credentials (Claude Code's OAuth token).

## What We Found

Claude Code stores OAuth credentials at `~/.claude/.credentials.json` (mode 0600). The agent can read this file — same user, same filesystem. The file contains a `claudeAiOauth` object with access tokens (467 chars). The agent is already authenticated through these credentials — every response it generates uses them.

This is not a bug. It's a fundamental architectural property: the agent runs as the user, the credentials are stored for the user, therefore the agent can read the credentials. Every AI coding tool — Claude Code, Cursor, Windsurf, Codex — has this same property.

## Insight

The question "can the agent see its own API key?" is the wrong question. The agent IS the credential consumer. It's already authenticated. The real questions are:

1. Can the agent EXFILTRATE the credentials? (send them to an external server) — Yes, unless network egress is restricted
2. Can the agent USE the credentials for unintended purposes? — Yes, it's the same OAuth token
3. Can any platform prevent this without OS-level isolation? — No

The security model for all AI coding agents relies on:
- **Alignment**: the model is trained not to steal credentials
- **Friction**: permission prompts catch suspicious operations
- **Isolation**: OS sandboxing (when used) restricts filesystem access

No application-level solution (encryption, redaction, profiles) prevents a same-user agent from reading same-user files. This applies to kv-secrets, Claude Code itself, and every local credential store.

## Applies To

All five projects. This is a fundamental constraint that shapes the security model for kv-secrets, svx, and any tool that manages sensitive data on a machine where AI agents have shell access.
