# Satish values closed loops — open-ended reports frustrate him

## Context

Multiple interactions across sessions. Pattern emerged.

## Observation

Satish consistently pushes back when I present analysis without action. "So what?" is his implicit question every time. He doesn't want a description of the problem — he wants the problem identified, the fix proposed, and the fix implemented. Open loops (problems stated but not resolved) are actively frustrating to him.

This applies to:
- Code reviews: don't just list issues, fix them
- Research: don't just summarize findings, apply them
- Tools: don't just describe what they could do, build them and prove they work

## Significance

This is a working style preference that should inform every interaction. Never present an unresolved problem without either (a) a proposed solution or (b) an explicit "I need your input on X before I can proceed." The loop must close or the blocker must be named.
