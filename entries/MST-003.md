# Tests don't verify the CLI actually works end-to-end

## Context

Built 7 CLI commands but only tested the underlying modules, not the CLI itself. Tested parser, validator, graph, query, health separately — but never tested that `python3 -m tools health` actually produces correct output.

## What Went Wrong

This is an integration gap. Each module is tested in isolation, but the CLI (which is the user-facing interface) has zero automated tests. I manually tested it, which is fine for now but won't scale.

If someone changes the parser's return signature, the CLI would break silently because no test exercises the full path: files on disk → parser → validator → health score → formatted output.

## Root Cause

I treated CLI as "just wiring" and didn't think it needed tests. But the wiring IS where integration bugs live.

## Prevention

Add at least one end-to-end test per CLI command that:
1. Creates real entry files in a temp directory
2. Runs the full pipeline (parse → analyze → format)
3. Verifies the output contains expected content

Not 50 tests — one per command. 8 tests that exercise the real path. This is tool limitation #2.
