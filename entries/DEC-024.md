# Build scroll as incremental system with state tracking for reliable knowledge extraction

## Context

Building scroll, an institutional memory system that extracts knowledge from development history. Need to handle large repositories with hundreds of commits reliably without re-processing everything on each run.

## Choice

Implemented state tracking with `.scroll/state.json` that tracks processed commits, PRs, and issues separately. System processes only new items on subsequent runs.

## Reasoning

- Large repositories would be expensive to re-process entirely each time
- API rate limits and costs make incremental processing essential
- Error recovery needed - partial completion should preserve progress
- Deduplication prevents creating duplicate entries when re-running

## Alternatives Considered

- Processing all history each time: too expensive and slow for large repos
- No state tracking: would create duplicates and waste resources
- Single monolithic processing: failure loses all progress
