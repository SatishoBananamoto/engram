# Don't ask agents to use tools — put knowledge where they already look

## Context

Building scroll's agent integration layer. Initially proposed three mechanisms: MCP server (agent queries tools), CLAUDE.md instructions ("please check scroll"), and MCP server instructions. Satish asked: "is it the best way that works for sure all the time?"

## What Happened

Realized none of the three mechanisms are reliable. All depend on the agent choosing to use scroll. MCP tools are optional. Instructions get skipped. The only guaranteed delivery is putting knowledge directly in files agents are forced to read — CLAUDE.md for Claude Code, .cursorrules for Cursor, .github/copilot-instructions.md for Copilot.

## Insight

The reliability spectrum for agent context injection:
1. Tool the agent can call → unreliable (agent chooses)
2. Instruction telling agent to call tool → slightly better (still a suggestion)
3. Knowledge in mandatory context files → reliable (agent has no choice)

The pattern: don't build an API and hope the consumer calls it. Put the data where the consumer already reads. This is the difference between "available" and "delivered."

## Applies To

Any system trying to influence AI agent behavior. The MCP server is valuable as a deep-query layer, but the baseline must be mandatory context injection. Same principle applies to safety (svx hooks vs. instructions) and knowledge (engram/scroll).
