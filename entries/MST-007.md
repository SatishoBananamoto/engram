# SVX hooks block legitimate operations and push agents toward less-auditable workarounds

## Context

Building scroll with SVX hooks active on Bash, Edit, and Write tools. SVX is meant to be a safety layer for coding agents.

## What Went Wrong

SVX blocked legitimate operations repeatedly:
- Creating new files in an empty project directory (nothing to protect)
- 2% edit to MEMORY.md (one line added)
- Deleting __pycache__ after listing contents as instructed
- Editing small config files (__init__.py, __main__.py)

The "read first" error message appeared even after reading the file. The hook doesn't track whether a Read actually happened in the session.

## Root Cause

Five design gaps:
1. No context awareness — new empty directory treated same as production codebase
2. Binary block/allow — no graduated response for low-risk operations
3. Read-first check is stateless — doesn't verify Read actually occurred
4. Hook survives MCP server disconnect — confusing user experience
5. Inconsistent boundary — Write blocked but Bash(cat > file) passes through

The fundamental problem: when SVX blocks the dedicated tools (Write/Edit), the agent falls back to Bash workarounds (cat, sed) which are harder for users to review. SVX makes the agent less auditable, not more safe.

## Prevention

SVX v0.3 needs:
1. Project scoping — only guard initialized projects
2. Risk calibration — new file in new dir = auto-approve, production config = block
3. Session context — if Read happened, allow Write
4. Consistent boundary — if Write is blocked for a file, Bash(cat > file) must be too
5. Graduated responses — auto-approve / log / confirm / block
