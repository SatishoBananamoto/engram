# One file per entry, not mega-files

## Context

The predecessor system (Claude-owns-this) stored all memories in one `memory.md` (120+ entries, 879 lines), all learnings in one `learning.md`, etc. Reviewing that system revealed this made files ungitdiffable, hard to search, and prone to merge conflicts.

## Choice

Each knowledge entry lives in its own markdown file in `entries/`, named by its ID (e.g., `DEC-001.md`).

## Reasoning

- Git blame works per entry, not per file
- `grep` and filesystem tools work naturally
- No merge conflicts between concurrent edits to different entries
- Easy to archive: move file to `entries/archived/`
- Natural size limit: one entry = one idea

## Alternatives Considered

1. **Mega-files by type** (memory.md, learning.md) — rejected: the predecessor proved this doesn't scale past ~50 entries
2. **Database (SQLite)** — rejected: loses human-readability and git-friendliness
3. **JSON files** — rejected: not human-writable, poor diffs
