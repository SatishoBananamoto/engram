# Add domain/subdomain taxonomy to engram entries

## Context

Agents working across projects need structured discovery. Freeform tags are inconsistent — one agent writes "security," another writes "auth," another writes "access-control." Cross-project queries fail because there's no shared vocabulary.

## Choice

Add `domain` and `subdomain` fields to engram entries. Seven fixed domains: security, architecture, performance, testing, ux, devops, strategy. Each with 3-5 predefined subdomains. Entries get exactly one domain and one subdomain.

## Reasoning

- Agents can query `domain:security` and find ALL security learnings across all projects
- Graph clustering becomes meaningful (domain = natural cluster boundary)
- The taxonomy is small enough (7 domains) that agents can remember it without looking it up
- Freeform tags remain for additional context, but domain/subdomain is the structured layer

## Alternatives Considered

- Tags only (current): inconsistent, no shared vocabulary across sessions
- Hierarchical categories (3+ levels): too complex, agents won't use correctly
- Auto-classification by LLM: adds latency and uncertainty to entry creation

## Applies To

All projects. This is an engram schema change that benefits cross-project knowledge flow.
