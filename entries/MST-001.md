# Parser silently accepted entries with missing tags

## Context

Building the parser. Tags are a required field per the schema.

## What Went Wrong

First version of `parse_entry()` treated missing tags the same as missing optional fields — it returned an entry with an empty tag list and no error. An entry with no tags is a schema violation but the parser didn't report it.

## Root Cause

I made `tags` default to `[]` in the Entry dataclass and didn't add explicit validation for "tags list is non-empty." The code path for required-field checking only covered id, type, and date. Tags fell through because they had a default.

## Prevention

Required-field validation must check for *semantic emptiness*, not just *key presence*. An empty list `[]` is present but empty. The fix: after extracting tags, check `if not tags:` and add a ParseError. This is now tested by `test_missing_tags` in test_parser.py.
