# Proxy/daemon model is the emerging standard for AI agent credentials

## Context

Researched the full competitive landscape for AI agent credential security (2026-03-26). Three models exist:

1. **SDK Retrieval** (1Password, Vault, Doppler): Agent gets plaintext secret. Vulnerable to prompt injection — once the key is in agent memory, it's compromised.
2. **Proxy/Daemon** (kv-secrets, AgentSecrets, OneCLI): Agent sends request with key NAME. Proxy resolves real credential, injects into HTTP request. Key never enters agent memory.
3. **Signing-only** (ssh-agent, gpg-agent): Key can only sign, never be exported. kv_api is this model applied to API calls.

## Key Findings

- kv_api implements the ssh-agent security model for API keys. The daemon makes the call, the key never leaves. Provider registry constrains destinations (like ssh-agent destination constraints).
- AgentSecrets (Go, HTTP proxy, HN launch March 2026) and OneCLI (Rust, HTTP gateway, HN launch March 2026) do the same thing. kv was first but the space is converging.
- 1Password launched "Unified Access" with Anthropic partnership (March 2026). Enterprise/cloud approach. Their SDK model is WEAKER — `secrets.resolve()` returns plaintext to agent memory.
- kv's differentiators: MCP-native, passphrase+TOTP interface barrier, kv_run for legacy apps, zero dependencies.
- kv's gaps vs competitors: no SSRF protection, no audit logging, hardcoded provider registry (5 providers), no cloud sync shipped.

## Why This Matters

The proxy model validates kv's core architecture. But the space is moving fast — table stakes are rising (audit logs, dynamic providers, SSRF protection). kv's MCP-native integration is a moat that HTTP-proxy competitors don't have.
