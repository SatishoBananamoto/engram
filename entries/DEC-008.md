# PyPI-only for transitive dependency analysis, not GitHub

## Context

Cascade risk needs to analyze transitive dependencies. Two approaches: use GitHub API (rich signals — contributors, commit activity, issues) or PyPI API (limited signals — releases, yank rate).

## Choice

PyPI-only for transitive deps. GitHub for direct deps only.

## Reasoning

GitHub API costs ~7 calls per package (repo info, contributors, commit activity, issues, community profile, funding, security policy). A project with 20 direct deps × 5 transitive each = 700 GitHub calls. Even authenticated (5000/hr), this exhausts the budget scanning a single project. PyPI has no rate limit.

The tradeoff: less signal depth per transitive dep, but you see the WHOLE dependency tree cheaply. Deep analysis on direct deps (GitHub), breadth analysis on transitive (PyPI). A risky transitive dep at depth 3 matters less than a risky direct dep anyway (depth decay: 1.0 / 0.7 / 0.4).

## Alternatives Considered

- **GitHub for everything**: Rate limit wall after 3-4 packages with transitive deps. Unusable.
- **Hybrid (GitHub for depth-1 transitive only)**: Complex, still expensive, marginal improvement.
- **Cached GitHub data**: Adds disk cache complexity. Doesn't solve first-run problem.

## Applies To

Vigil only. But the principle — trade depth for breadth when API budget is constrained — applies to any tool doing multi-layer API analysis.
