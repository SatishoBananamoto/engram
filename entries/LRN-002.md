# 6 modules is enough when each one earns its place

## Context

Built engram with 6 Python modules: parser, validator, graph, query, health, cli. The predecessor had 70+.

## What Happened

Every module was built because it was needed, not because it might be useful. Parser exists because entries need reading. Validator exists because broken links need catching. Graph exists because connections need analyzing. Query exists because entries need finding. Health exists because the system needs a single "am I ok?" number. CLI exists because humans need an interface.

No module was built speculatively. No "it might be useful later."

## Insight

The right number of modules is the number where removing any one would lose a capability you actually use. The predecessor's 70 modules included wavelets, epidemic models, and Fiedler vectors — none of which informed any decision about the knowledge base. They were built because building is easier than deciding what not to build.

## Applies To

Every project. Before adding a module, ask: "What decision or action does this enable that I can't do without it?" If the answer is nothing concrete, don't build it.
