# Vigil rate limiting — config + auth + budget tracking approach

## Decision

Implemented rate limiting fix for vigil as three layers:

1. **Config module** (`config.py`): `~/.config/vigil/config.toml` with XDG support, 600 permissions for token security. Token resolution priority: explicit param → `GITHUB_TOKEN` env var → config file.

2. **Auth CLI** (`vigil auth login/status/logout`): Guided token setup with validation (hits `/rate_limit` to verify), saves to config. Status command shows remaining budget and scannable package count.

3. **Budget tracking in GitHubClient**: Tracks `remaining`, `limit`, `requests_made` from response headers. Pre-flight check before scanning warns if budget is insufficient. Mid-scan `RateLimitError` now caught cleanly with partial results displayed.

## Why not other approaches

- **No disk-persistent cache**: 1hr in-memory TTL is sufficient for a single scan session. Disk cache adds complexity for marginal gain at this stage.
- **No async/concurrent fetching**: Sequential is fine for now. Async is a Phase 3 concern when scanning 100+ packages.
- **No retry/backoff**: Rate limits are hard walls, not transient failures. Waiting and retrying would block the terminal. Better to show partial results and tell the user when to come back.

## API call budget

~7 calls per package (repo, commit_activity, contributors, issues, community/profile, users/owner, contents/FUNDING.yml). Contributors call is cached across MaintainerAnalyzer and CommunityAnalyzer. At 60 req/hr unauth → ~8 packages. At 5000 req/hr auth → ~714 packages.
