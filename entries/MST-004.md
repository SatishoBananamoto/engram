# Validator doesn't report that DEC-002 was superseded but DEC-004 exists — it only checks one direction

## Context

DEC-004 supersedes DEC-002. I manually changed DEC-002 to `status: superseded`. The validator checks if the superseded entry's status is correct — but what if I forgot to change it?

## What Went Wrong

The validator's `check_supersession` only checks: "if entry X says it supersedes Y, is Y marked as superseded?" It doesn't check the reverse: "if entry Y is marked superseded, does some entry claim to supersede it?"

This means:
- If I mark DEC-002 as superseded but forget to create DEC-004 with `supersedes: DEC-002`, the system would have a superseded entry with no replacement. That's an orphaned decision.
- The validator wouldn't catch this.

## Root Cause

One-directional checking. The supersession chain should be validated in both directions.

## Prevention

Add a reverse check: for every entry with `status: superseded`, verify that at least one other entry has `supersedes: <that-id>`. If not, warn: "superseded entry with no replacement."
