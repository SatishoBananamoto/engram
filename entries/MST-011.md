# Output redaction doesn't prevent filesystem exfiltration

## Context

During adversarial testing of kv-secrets' agent daemon, we discovered that output redaction on the socket channel is insufficient. The subprocess runs with secrets as environment variables and can write them to files.

## What Went Wrong

The daemon redacts stdout/stderr before returning through the Unix socket — `printenv` returns `[REDACTED]`. But the subprocess can write to the filesystem directly:

```bash
printenv > /tmp/leaked_secrets.txt
```

The agent then reads the file, bypassing socket redaction entirely. All 5 API keys were exposed.

## Root Cause

Redaction operates on the communication channel (socket output), not on the execution environment. The subprocess has full env vars AND full filesystem write access. Any process with both can exfiltrate secrets to disk.

## Prevention

This is a fundamental same-user, same-machine limitation. No software fix prevents a subprocess from writing its own environment to a file without OS-level sandboxing. The practical mitigations:

1. Accept as known limitation for deliberate/adversarial attacks
2. Output redaction still catches accidental leaks (error messages containing secrets)
3. Post-execution file monitoring could detect suspicious writes (defense-in-depth, not prevention)
4. True containment requires running subprocess in a sandbox (container, seccomp, namespace)

## Applies To

Any system that injects secrets as env vars into subprocesses — not just kv-secrets. This is the same limitation ssh-agent, docker secrets, and Kubernetes secrets face when mounted as env vars.
