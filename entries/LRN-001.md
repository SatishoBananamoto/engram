# Functional tests catch more than syntax tests

## Context

Building the parser and validator for engram. Had to decide what "tested" means.

## What Happened

Wrote 57 tests across parser and validator. The most valuable tests aren't the ones that verify parsing works — those are table stakes. The valuable ones are:

- `test_catches_broken_link`: Creates an entry linking to a nonexistent ID. Verifies the validator flags it.
- `test_catches_unsuperseded_old_entry`: Creates an entry that supersedes another, but the old one is still marked active. Catches the inconsistency.
- `test_id_prefix_mismatch`: Entry has `id: LRN-001` but `type: decision`. Catches schema violation.

These test *behavior*, not *code paths*.

## Insight

The question isn't "does `parse_entry()` return an Entry?" — it's "if I give it broken data, does it tell me what's broken and where?" Testing the failure paths is where the real validation lives.

## Applies To

Every module in this system. Before writing tests, ask: "What should this module catch that a human would miss?" Then test exactly that.
